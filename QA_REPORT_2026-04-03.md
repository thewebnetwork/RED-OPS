## PHASE 2 — Bug Hunt

### Fixed
- **Admin role DB queries** (CRITICAL): `feedback.py`, `sla_monitor.py`, `workflow_engine.py` searched for `{"role": "Admin"}` in the database, but users are stored with `role: "Administrator"`. These queries returned zero results, meaning admin notifications for feedback, SLA breaches, and workflow events were silently failing. Fixed to `{"role": {"$in": ["Administrator", "Admin"]}}`.

- **Debug print statements**: Replaced 6 `print()` calls in `email.py` (SMTP mock fallback) with a single `logger.warning()` line.

### Already Clean
- **Frontend console.log**: 0 instances found in pages/ and components/
- **Frontend role checks**: All use `|| user?.role === 'Admin'` pattern alongside `'Administrator'` — correct and consistent
- **Backend `require_roles(["Admin"])`**: Works correctly via `ROLE_ALIASES` mapping in `auth.py` that maps "Admin" → "Administrator"

### Noted (Low Risk)
- `server_legacy.py` still has hardcoded `{"role": "Admin"}` queries (5 instances) — this file is unused in production (server_v2.py is the active server)
- Some user queries in tasks.py, teams.py, dashboard.py fetch without explicit `"password": 0` projection — password is excluded at the response model level via Pydantic, so no actual leakage occurs

## PHASE 3 — Security

### Verified
- All mutating endpoints have `Depends(require_roles([...]))` or `Depends(get_current_user)` guards
- JWT tokens stored in localStorage (acceptable for this app's threat model)
- CORS origins in `server_v2.py` are explicitly whitelisted (redops.redribbongroup.ca, red-ops.vercel.app) + runtime env var — NOT wildcard
- Password field excluded from all API responses via Pydantic model exclusion + `{"password": 0}` on list queries
- Admin seed in server_v2.py now requires `ADMIN_PASSWORD` env var (no hardcoded fallback)
