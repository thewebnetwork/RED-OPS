### 4.2 Top 3 architectural decisions needed (can't be patched)

1. **Pick ONE org-resolution mechanism.** Today there are two (the `primary_org_id`+`org_members` membership lookup in `get_current_user`, and the in-route `org_id or team_id or id` fallback). Either commit to true multi-tenancy (every user must belong to an org, fail closed if not) or commit to single-tenant with `user.id` as the de-facto org. The current hybrid means every route author has to remember the fallback or it leaks (cf. `documents.py`). Until this is decided, every new route is a roulette spin.
2. **Decide the Workflows feature's fate.** The engine runs (`workflow_engine.py`, `sla_policy_engine.py`, `routes/workflows.py` registered at `server_v2.py:292`) but admins can't see or edit anything. Either restore the UI (and accept the maintenance load) or rip the backend out. The middle state is silent automation that nobody can audit or change.
3. **Decide on observability strategy.** No Sentry, no global exception handler, swallowed frontend errors, polling-based dashboards. Production-readiness requires a coherent decision: log aggregator, error tracker, uptime monitor. Without one, every production bug is a user complaint.

### 4.4 Recommended fix sequence (so later work doesn't break earlier)

1. **Stop the bleeding.** Apply fix #1 (`documents.py` org scoping) before anything else. Single PR.
2. **Eliminate the latent footguns.** Fixes #2 + #3 + #5 + #8 — purely additive cleanups; no regression risk.
3. **Decide architecture.** Architectural decisions §4.2.1 and §4.2.2. Document the call in `CLAUDE.md` so the next pass has a north star.
4. **Sweep RBAC consistency** (fix #4) — once the canonical role names are pinned, the helper write-up is safe.
5. **Doc reconciliation** (fix #10) before any feature work — otherwise every new contributor will absorb stale assumptions and reproduce the same drift.
6. **Dead code purge** (fix #9). After §3 architecture decisions are made, the orphan list may shrink (e.g., if Workflows UI is restored).
7. **Observability** (architectural decision §4.2.3 → implementation). Sentry-or-equivalent first, structured logging second.

## Decision Log

### 2026-04-17: Org-resolution mechanism (§4.2.1)

**Chose:** Single-tenant-per-user. Every user is their own org. `org_id = user.id`.

**Rationale:** RRM serves individual realtors; no team customers exist. The dual mechanism (primary_org_id + org_members lookup in get_current_user, plus 3-level fallback in every route) created a class of cross-tenant leak bugs (see §1.1 documents.py). Single-tenant eliminates this class entirely — the failure mode becomes empty screen, not data leak.

**Implementation:** Branch `fix/single-tenant-migration`. Changes:
- `utils/auth.py`: `get_current_user` sets `org_id = user["id"]` unconditionally
- `utils/tenancy.py`: new shared `resolve_org_id()` helper (returns `user["id"]`)
- 24 route/service files: replaced 3-level fallback with `resolve_org_id()` calls
- `organizations.py`: marked deprecated (deletion in separate pass)
- `OrgContext.js`: short-circuited to `user.id` (cleanup in separate pass)
- Data migration plan documented at `backend/scripts/migrations/PENDING_single_tenant_data_migration.md`
